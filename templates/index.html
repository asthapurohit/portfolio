HDFC MF FAQ Chatbot

Live

A RAG-powered chatbot that answers questions about HDFC Mutual Fund products using retrieval-augmented generation. Built to make dense financial documents conversational and accessible.

PM SIGNAL — The best interface for complex information is a conversation, not a document.

Financial products fail users not because they're complex — but because the complexity is never translated.

Python RAG Streamlit LLM
Problem HDFC MF documentation is dense and hard to navigate — users can't find answers without reading entire PDFs.
How I built it Chunked and embedded the FAQ documents, built a retrieval pipeline, wrapped it in a Streamlit interface users can query in plain English.
What I'd change I'd add source citations per answer so users can verify — trust is the core problem in financial information.
What I didn't build: A comparison feature across fund types — useful but out of scope for a first version.

GeoBehaviour

In Progress

A geo-analytics platform that decodes local consumer behavior using proxy signals. Identifies demand patterns that traditional surveys miss.

PM SIGNAL — Proxy signals > surveys when behavior is the data

Currently in early signal validation — mapping consumption patterns across urban zones.

Python SQL Maps API
Problem Traditional surveys capture stated intent, not real behavior — local demand signals were invisible to small businesses.
How I validated Mapped proxy signals (foot traffic patterns, search density, review velocity) across 3 urban zones to find demand gaps.
What I'd change I'd talk to 5 small business owners before building — I assumed the pain point without enough direct validation.
What I didn't build: A real-time alert system for demand spikes — technically interesting but not the core job-to-be-done yet.

ChatGPT Voice Adoption (Whisper Mode)

PRD

Identified why voice mode had low adoption among students despite high awareness. Designed a feature strategy addressing social friction, not discoverability.

PM SIGNAL — Adoption failures are social before they're UX

Voice fails not because users can't find it — but because using it in public feels socially unsafe.

PDF • Behavioral Product Design

Problem Voice mode awareness was high but usage near zero — the barrier was after discovery, not before it.
How I validated Mapped 12 student use-case scenarios — social context (library, class, public) correlated strongly with drop-off.
What I'd change I'd run intercept interviews before writing the PRD — I relied on proxy data more than I should have.
↑ 38% simulated task adoption in redesigned flow
What I didn't build: A "voice confidence score" feature — it added complexity without solving the core social friction.

Goodreads – From Tracking to Habit

PRD

Redesigned the reading log experience to capture intent at the moment it occurs, reducing drop-off in progress tracking through contextual nudges.

PM SIGNAL — Capture intent at the moment it occurs, not later

Users don't forget to log — they lose the moment of intent before they get to the app.

PDF • Habit Formation & Retention

Problem Users open Goodreads to log progress after reading — by then the moment of intent has passed and friction wins.
How I validated Audited the logging flow step by step, counting taps to complete a progress update — 7 steps for a 2-second task.
What I'd change I'd prototype the contextual nudge and test it with real readers before committing to the full feature spec.
↓ 7 steps → 2 steps to log progress
What I didn't build: A social reading feed — users asked for it but it would have made logging feel even more performative.

Goodreads Product Teardown

Teardown

Audited Goodreads across user flows, engagement loops, and retention mechanics. Mapped where the product loses users and why the core loop is broken.

PM SIGNAL — A broken core loop can't be fixed with features

Logging is treated as admin work — it sits outside the reading experience instead of inside it.

PDF • Product Analysis

Problem Goodreads has millions of users but retention is weak — I wanted to find exactly where and why the core loop breaks.
How I validated Walked every major user flow from onboarding to re-engagement, mapping where momentum drops and why.
What I'd change I'd quantify drop-off rates per flow step if I had access to real data — the teardown is qualitative where it should be mixed.
What I didn't build: A full redesign concept — a teardown should diagnose, not prescribe. That's what the PRD is for.
